What is the DQ Framework?
An automated, regulatory-grade Data Quality monitoring system built on Databricks. It continuously validates critical business data against a library of reusable rules aligned with BCBS239 (Basel III risk data aggregation principles) and Solvency II reporting requirements. Every validated dataset gets a pass/fail score per rule, an overall DQ percentage, a categorised breach register, and a live AI/BI dashboard — all refreshed automatically on a configurable schedule.

▶ DQ Pipeline - 4 Notebook Tasks per Entity

1 Setup 00_setup.py Creates registry tables, seeds rules and mappings, CTAS source Gold data. Entity-specific, generated from config.
2 Apply Rules 01_apply_dq_rules.py Evaluates every mapped rule row-by-row. Writes <entity>_dq with a DQ_RESULT string column. Shared notebook.
3 Aggregate 02_aggregate_dq_results.py Sums pass/fail counts per rule. Appends one row per rule to dq_results audit log. Shared notebook.
4 Dashboard 03_dq_dashboard.py Builds summary, category breakdown, per-rule detail, and breach register views. Shared notebook.

Tasks 2-4 are shared notebooks reused by every entity. Task 1 is generated automatically from the entity config file - no manual coding required.

📋 DQ Result Format

Each row in <entity>_dq retains all original columns plus a pipe-delimited DQ_RESULT string:

-- Example DQ_RESULT column value for one customer row: RULE_101: 1 | RULE_102: 1 | RULE_103: 0 | RULE_109: NULL | RULE_114: 1 | ... -- Meaning of each value: 1 = Rule passed for this row 0 = Rule failed - breach recorded in dq_results NULL = Rule not applicable (row excluded by status filter)
